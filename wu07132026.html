Overview

The project moved from a readable intake log to a working multi-stage communication and task-management system.

Supervisor submits the Housekeeping Form → source record and normalized JSON are created → notification email is sent → intake card appears → submission is acknowledged once → item is sorted into Work Queue or Recommendations Queue → status and notes are updated → completed work leaves the active queue → completed history remains available → completion email is sent once
The system now preserves both the original submission and the evolving working record. Source data remains unchanged while queue status, timestamps, notes, and email state accumulate separately.

Google Form and source-data preparation

  • Historical test entries were cleared from the working sheets.
  • A fresh test issue was submitted through the simplified Housekeeping Form.
  • The form-submit trigger produced the expected notification email.
  • Normalized JSON was stored successfully.
  • The form’s first question was simplified to identify the action type immediately.
  • The updated question wording What needs done? was added to the normalization patterns.
  • New submissions now map the selected task into the normalized issue field correctly.
Existing submissions retain the normalized JSON created at their original submission time. New normalization rules apply to new records unless a separate migration is run.

Recommendations_Queue

Sheet names were centralized in the project’s CONFIG object. A test function verified that Apps Script could find each queue by its exact name.

Intake viewer and sorting controls

  • Each intake card received a Send to Work Queue button.
  • Each intake card received a Send to Recommendations Queue button.
  • The buttons pass the correct submission ID.
  • Queue-writing functions create one persistent row in the selected queue.
  • Duplicate sorting is blocked.
  • The same Source ID cannot be placed in both queues accidentally.
  • LockService protects against competing browser tabs.
  • Sorted cards display a persistent queue label after refresh.
  • Sorting buttons disappear once the item has been handled.
Unsorted card → sorting buttons visible → user selects destination → working record created → viewer reloads → “Sorted to Work Queue” or “Sorted to Recommendations Queue” remains visible

Persistent acknowledgment

Acknowledgment was changed from a temporary browser-state effect into a persistent, one-time system event.

  • The system checks Acknowledgement_Log before sending.
  • The first click sends one acknowledgment email.
  • The acknowledgment event is logged.
  • The button changes to Acknowledged.
  • The state remains after refresh.
  • The button remains disabled on acknowledged records.
  • Duplicate acknowledgment emails are blocked at the backend.
The intended communication pattern is now one acknowledgment email when work is accepted and one completion email when the work is finished.

Active Work Queue viewer

A new Work Queue viewer was built and connected to ?view=work. It joins each working record to the original normalized source record by Source ID.

  • Pending and In Progress work appears as readable cards.
  • Original request details are retrieved from the source record.
  • Submitted and sorted timestamps are displayed.
  • Unit, room or area, description, request, and issue details are shown.
  • Notes JSON is parsed and displayed.
  • Missing source records produce a warning card instead of breaking the viewer.
  • An empty active queue displays No active tasks.

Work actions added

  • Start Work changes Pending to In Progress.
  • Started At is written only once.
  • Add Note appends timestamped JSON notes.
  • Earlier notes remain intact.
  • Complete writes Completed At and Updated At.
  • Pending tasks can be completed directly.
  • Completed tasks disappear from the active queue without being deleted.

Recommendations Queue viewer

A separate Recommendations Queue viewer was created for requests and ideas that require review rather than immediate work execution.

  • Unreviewed and Reviewed recommendations appear as cards.
  • Original request, suggestion, observation, unit, and room information are joined from the source record.
  • Mark Reviewed changes the status persistently.
  • Add Note stores timestamped JSON notes.
  • Archive removes the item from the active recommendations view.
  • Recommendations do not trigger work-completion emails.
Useful operational notes were already added, including the Meadows Housekeeping site URL, demonstrating that the queue can serve as an evolving reference record.

Completed Work viewer

A Completed Work history view was added and connected to ?view=completed.

  • Completed records appear once.
  • Active records are excluded.
  • Original request details remain available.
  • Sorted, Started, and Completed timestamps are displayed.
  • Duration is calculated when timestamps are available.
  • Notes remain readable.
  • Completion email state is visible.
  • Completed rows remain in Work_Queue as durable history.
Active Work Queue → Complete → status becomes Completed → active viewer filters the record out → Completed Work viewer displays the preserved record

Completion email safeguards

The completion workflow was extended so email delivery occurs as part of the successful Complete action rather than requiring another acknowledgment.

  • The completion function checks the current status.
  • The task is saved as Completed before email delivery is attempted.
  • The original requester email is used when available.
  • A configured fallback recipient remains available.
  • The message includes Source ID, completion time, request summary, and final note.
  • Completion Email Sent is set to TRUE after a successful send.
  • Repeated completion attempts do not send duplicate email.
  • A failed send leaves the email flag false for later retry.

Google Sites integration and navigation testing

The latest Apps Script deployment was embedded into the Meadows Housekeeping Google Site. The team verified that the updated viewer appeared after replacing the embedded URL.

  • The supervisor-facing site can display the intake viewer.
  • The active Work Queue, Recommendations Queue, and Completed Work views can be opened directly in a browser.
  • Navigation links between separate views were tested.
  • The browser version navigates correctly.
  • Repeated iframe navigation inside Google Sites was identified as unreliable.
Because repeated Apps Script page changes can cause the embedded tool to disappear, the system will be converted to a single-page viewer that changes content internally without changing the iframe URL.

Architecture decision: single-page dashboard

A conversion plan was created for replacing the separate HTML views with one Dashboard.html file.

One embedded Apps Script page → internal Intake button → internal Work Queue button → internal Recommendations button → internal Completed Work button → shared content area redraws → iframe remains loaded

The existing backend functions will remain in place:

This preserves the functioning data layer while consolidating only the front-end presentation and navigation.

Lesson 8 · Programmable Machine Control

From Relay Logic to PLC Control at the Automated Water Plant

The recycling center taught Monkey and Robot how relays, contactors, sensors, timers, and interlocks coordinate large machinery. At the automated water plant, the same control ideas have moved into a programmable logic controller. A transfer pump refuses to start automatically, one tank has nearly overflowed, and two identical pumps have accumulated very different operating hours.

Central question: How does a PLC observe physical conditions, make logical decisions, and command real machinery?

Key idea: A PLC repeatedly reads inputs, solves its program, updates outputs, performs internal checks, and begins again.

1. From relay logic to programmable logic

The control ideas from Lesson 7 remain, but many decisions now occur inside a controller.

Hardwired relay control

Physical contacts, relay coils, timers, and auxiliary contacts create the operating sequence.

  • Changing the sequence may require rewiring.
  • Each added function usually requires more hardware.
  • The wiring itself contains much of the logic.

PLC control

Input devices are wired to a controller, which evaluates a stored program and commands output devices.

  • Changing the sequence may require a program change.
  • Timers, counters, memory, and comparisons can be added in software.
  • Field wiring and program logic must agree.

Important continuity

A PLC does not erase the electrical principles from earlier lessons. It organizes and expands them.

Monkey & Robot

Monkey says, “The wires became a program.” Robot replies, “Some of the decision-making did. The motors, sensors, and power circuits are still real.”

2. The PLC as part of a larger control system

The PLC is the controller, but it depends on many other components.

Power supply

Provides the low-voltage power required by the processor and modules.

Processor

Stores and executes the control program.

Input modules

Receive electrical evidence from field devices.

Output modules

Command relays, contactors, lamps, valves, and other devices.

3. Inputs: how the physical world enters the program

Inputs convert field conditions into electrical and logical evidence.

Discrete inputs

Represent two general states: on or off, open or closed, true or false.

  • Float switch
  • Pressure switch
  • Emergency-stop contact
  • Overload contact

Analog inputs

Represent a changing measured value over a range.

  • Tank level
  • Pressure
  • Flow rate
  • Temperature

Input evidence

The PLC sees voltage, current, or data—not the physical process directly.

Question

Does an active PLC input prove that the tank is actually full?

No. It proves that the controller is receiving the input state associated with “full.” The sensor, wiring, address, and interpretation must still be correct.

Source evidence

A field input is a source record from the physical process. The program interprets that source and derives operating decisions.

4. Outputs: how the PLC affects machinery

Outputs convert program decisions into electrical commands.

Motor commands

A PLC output may energize an interposing relay or motor-starter coil.

Valve commands

A solenoid output can shift a valve or command a motorized actuator.

Status and alarms

Outputs may operate lamps, horns, beacons, or communication signals.

PLC output
Interposing relay
Contactor coil
Power contacts
Pump motor

Power limitation

A PLC output normally does not carry the operating current of a large motor.

Command is not proof

An energized output means the PLC issued a command. It does not prove that the final device moved.

5. Addresses, tags, and names

The program must identify each physical and internal signal.

Field device or condition Example address Descriptive tag Meaning
Low-level float I:0/0 Tank_Low Tank needs filling.
High-level float I:0/1 Tank_High Normal fill stop point reached.
Pump overload contact I:0/2 Pump1_OL_OK Motor protection circuit is healthy.
Pump contactor command O:0/0 Pump1_Run PLC is requesting Pump 1 operation.
Internal failure memory B3:0/0 Pump1_Failed Program has recorded a pump fault.

Question

What happens if the float is wired to I:0/3 but the program expects I:0/1?

The physical switch may work, but the program will be looking at the wrong input and may behave as though the signal never changed.

Documentation matters

Drawings, wire labels, input lists, output lists, and program tags must describe the same system.

6. The PLC scan cycle

The processor repeats a rapid sequence of observation, decision, and action.

1. Read inputs

The PLC records the current state of its input points.

2. Solve logic

The program is evaluated in its defined order.

3. Update outputs

Output states are changed according to the solved program.

4. Housekeeping

The controller performs diagnostics, communications, and internal tasks.

Question

If an input changes during program execution, when might ordinary logic recognize it?

Often during the next input-read portion of the scan, depending on controller design and instruction behavior.

Time matters

Even fast controllers operate in sequence. Scan time becomes important when signals change rapidly or events require precise timing.

7. PLC ladder instructions

PLC ladder logic resembles relay diagrams, but its symbols test and change stored states.

Examine-if-closed

True when the referenced bit is on.

Examine-if-open

True when the referenced bit is off.

Output energize

Controls an output or memory bit according to rung continuity.

Do not confuse symbol with hardware

A PLC instruction does not always describe whether the physical field device is mechanically normally open or normally closed.

What the instruction does

It tests the current logical value stored at a particular address or tag.

Question

Why can a PLC “contact” exist even when no physical contact exists?

Because the instruction may reference an internal memory bit, timer status, comparison result, output state, or communicated value.

Monkey & Robot

Monkey says, “I see a contact.” Robot replies, “You see an instruction asking a question about stored data.”

8. Automatic tank-filling sequence

A simple fill cycle combines permissives, start conditions, stop conditions, and feedback.

Start conditions

  • Tank is below low level
  • Automatic mode selected
  • Emergency-stop chain healthy
  • Overload healthy

Run conditions

  • Pump request remains active
  • High level not reached
  • No trip condition appears
  • Flow is proven

Stop conditions

  • High-level switch activates
  • Operator selects Off
  • Overload trips
  • Flow fails

Question

Why should the pump not rely only on the low-level switch?

The pump also needs a normal stopping point, safety permissives, equipment protection, and proof that the commanded flow actually occurs.

Process sequence

The low-level switch requests action. The high-level switch ends the normal cycle. Other conditions determine whether operation is permitted.

9. Timers inside a PLC

Software timers coordinate delays, proof windows, warnings, and fault responses.

Preset

The target time value.

Accumulated value

The time already measured.

Timing status

Indicates that the timer is actively counting.

Done status

Indicates that the preset has been reached.

Flow-proving example

  1. PLC commands Pump 1.
  2. A five-second timer begins.
  3. Flow switch must activate before the timer finishes.
  4. If flow is proven, operation continues.
  5. If flow is not proven, the pump stops and a fault is recorded.

Retentive timing

A retentive timer may preserve accumulated operating time so maintenance can track how long a pump has run.

Limited retries

A failed pump should not restart endlessly. The logic can limit attempts and require operator attention.

Command and proof

The run output is a command. The flow switch is evidence that the commanded process actually began.

10. Lead-lag pump control

Two pumps can provide redundancy, increased capacity, and balanced wear.

Lead pump

Starts first during normal demand.

Lag pump

Starts if demand remains high or the lead pump fails.

Alternation

The PLC periodically changes which pump is assigned as lead.

Question

Why might two identical pumps show very different runtime totals?

The lead assignment may never alternate, one pump may be unavailable, the runtime logic may be incorrect, or the counters may not be tied to proven operation.

Maintenance value

Balanced runtime can reduce uneven wear and helps maintenance teams compare equipment performance more fairly.

11. Hand, Off, and Automatic modes

Operating modes determine where a run request comes from.

Hand

The operator directly requests operation, subject to required protective conditions.

Off

The equipment is prevented from running.

Automatic

The PLC starts and stops the equipment according to process conditions.

Question

Should Hand mode bypass the motor overload or emergency stop?

No. Manual control changes the source of the run request; it should not defeat essential protection.

Manual is not unprotected

Hand mode is for direct operation, not for ignoring hazardous or damaging conditions.

12. Internal memory and system state

Not every PLC value comes directly from a physical input.

Cycle state

Records whether an automatic cycle is inactive, filling, proving flow, complete, or faulted.

Fault memory

Records that an abnormal event occurred even after the original condition disappears.

Selection memory

Stores which pump is currently assigned as lead.

Source and derived state

Tank_High may come from a field switch. Pump1_Failed may be a derived conclusion based on several pieces of evidence.

Memory builds sequence

Internal bits allow the controller to remember what has happened and determine what should happen next.

13. Alarms, trips, and acknowledgment

Different abnormal conditions require different responses.

Warning

Attention is needed, but operation may continue.

Trip

Permission to operate is removed.

Latched alarm

The event remains recorded until acknowledged or reset.

Current condition

Shows whether the abnormal condition is still physically present.

Example alarm levels

  • High level: operator warning
  • High-high level: pump shutdown and latched alarm
  • No flow after start: pump trip
  • Repeated pump failure: automatic lockout

Acknowledgment versus reset

Acknowledgment confirms that an operator has seen the alarm. Reset attempts to clear a stored fault after the cause has been corrected.

14. Ordinary PLC logic and safety functions

Routine control logic should not be confused with safety-rated protection.

Ordinary control

Coordinates process operation, sequencing, alarms, and production decisions.

Safety control

May require safety relays, redundant channels, monitored contacts, or a safety-rated PLC.

Deliberate restart

After a critical safety fault, the system should not restart unexpectedly.

Essential rule

Software convenience must never replace required electrical and mechanical safeguards.

Lesson 7 connection

Normally closed safety chains, monitored devices, overloads, and deliberate reset remain important in PLC-controlled systems.

15. Troubleshooting the pump that will not start

The tank is low, Automatic mode is selected, and the program appears to request Pump 1.

Trace the input and logic path

  1. Observe the actual tank condition.
  2. Check the low-level device.
  3. Check voltage at the PLC input.
  4. Check the input indicator.
  5. Check the input tag in the program.
  6. Trace every permissive on the rung.
  7. Confirm the output instruction becomes true.

Trace the physical output path

  1. Check the PLC output indicator.
  2. Check voltage at the output terminal.
  3. Check the interposing relay.
  4. Check the contactor coil circuit.
  5. Check the overload and contactor.
  6. Check motor power.
  7. Check the pump and mechanical load.

Question

What is the final pump fault?

A loose wire between the interposing relay and the contactor coil prevents the PLC command from reaching the motor starter.

Second fault

Why did the tank nearly overfill?

The high-level float was wired to a different PLC input than the program expected, so the physical device changed but the intended program tag did not.

Monkey & Robot

Monkey says, “The output light was on, so I thought the pump was running.” Robot replies, “A command is evidence of intention, not evidence of motion.”

Core diagnostic habit

Trace the complete chain: physical condition, field device, input wiring, stored value, program decision, output wiring, final control device, and actual motion.

16. Lesson summary

The central ideas behind programmable industrial control.

Key takeaways

  • PLCs read input signals and command outputs.
  • The PLC operates as part of a larger electrical and mechanical system.
  • Relay ladder logic and PLC ladder logic share important ideas.
  • PLC instructions test stored logical states, not necessarily physical contacts.
  • The processor repeatedly reads inputs, solves logic, updates outputs, and performs internal tasks.
  • Discrete inputs represent two-state conditions; analog inputs represent changing measured values.
  • Outputs usually command relays, contactors, valves, lamps, and alarms.
  • Tags and addresses must match the actual field wiring.
  • Timers create delays, proof windows, and fault responses.
  • Internal memory records cycle state, selections, and faults.
  • Lead-lag control provides redundancy and balances equipment wear.
  • Manual mode should not bypass essential protection.
  • Acknowledgment and reset are different actions.
  • Routine PLC logic is not automatically safety-rated logic.
  • An output command does not prove that physical machinery operated.
  • Troubleshooting must follow the complete path from field condition to actual machine response.

Final Check

What are the three main operating steps of a PLC scan?

Read inputs, solve the program, and update outputs, followed by controller housekeeping and repetition.

Final Check

What is the main troubleshooting lesson?

Verify each stage of the evidence-and-command chain instead of assuming that a PLC indication proves physical operation.

# Structured Problem Solving and Goal Tracking Ruleset

**Version:** 1.0  
**Date:** July 16, 2026  
**Status:** Initial working version

## Purpose

This ruleset converts a confusing real-world problem or goal into a documented, testable case with:

- a clearly stated need,
- a defined outcome,
- preserved evidence and context,
- a map of the existing process,
- testable hypotheses,
- an ordered roadmap,
- observable test points,
- written communication records,
- escalation paths,
- and verified closure criteria.

The framework is intended to evolve quickly through real use.

## Core Principle

> Begin with the user’s actual need, preserve the context, and make every search, message, and action serve a defined outcome.

## Intellectual Foundations

The framework combines three useful traditions.

### Scientific method

Observe the situation, define the question, develop hypotheses, choose actions that produce evidence, evaluate results, and revise the model.

### Problem-solving therapy

Clarify the problem, generate options, evaluate practical tradeoffs, choose an achievable plan, carry it out, and review the outcome.

### Case management

Preserve the history, identify ownership, track handoffs, maintain next actions, escalate when progress stops, and close only when the original need has actually been met.

## Workflow

### 1. Capture

Record the triggering event before details are lost:

- What was the user trying to accomplish?
- What was expected?
- What actually happened?
- What consequence followed?
- What evidence exists?

### 2. Formulate

Separate the symptom from the actual operational problem.

Use:

> I am trying to accomplish **X**, but **Y** is preventing it, resulting in **Z**.

### 3. Define Success

State:

- minimum acceptable resolution,
- ideal resolution,
- observable closure criteria.

### 4. Inventory Context

Separate:

- confirmed facts,
- unknowns,
- assumptions,
- constraints,
- communication preferences,
- accessibility needs.

Search existing records before asking the user to repeat information.

### 5. Map the Process

Describe:

> Entry point → requirements → actions → handoffs → failure point → fallback → outcome

Mark loops, dead ends, unclear ownership, and places where context is lost.

### 6. Develop Hypotheses

List plausible explanations without presenting them as facts. For each one, identify evidence that would support or weaken it.

### 7. Generate Options

Create more than one practical path when possible. Compare:

- effort,
- stress,
- cost,
- time,
- reversibility,
- risk,
- likelihood of success.

### 8. Build the Roadmap

Every step must include:

- action,
- purpose,
- required inputs,
- expected output,
- success condition,
- test point,
- fallback if it fails.

### 9. Execute and Record

Prepare communications from the case record. Preserve dates, recipients, case numbers, attachments, responses, and status changes.

### 10. Test

At each checkpoint, assign one result:

- **Pass:** the condition was met; continue.
- **Partial:** progress occurred but more information or action is required.
- **Fail:** the condition was not met; use the planned fallback or escalation.

### 11. Escalate or Revise

When progress stops:

- identify the type of failure,
- revise the hypothesis if necessary,
- preserve the existing case history,
- request transfer rather than restart,
- seek accountable ownership.

### 12. Close and Learn

Close only after the original goal has been achieved and tested, or when the user explicitly chooses to stop.

Record:

- final resolution,
- unresolved limitations,
- lessons learned,
- proposed ruleset changes.

## Essential Case Fields

A case should contain:

- Case ID
- Title
- Type
- Status
- Priority
- Triggering event
- User goal
- Problem statement
- Desired outcome
- Constraints
- Communication preferences
- Known facts
- Unknowns
- Assumptions
- Hypotheses
- Evidence
- Stakeholders
- Process map
- Barriers
- Roadmap
- Test points
- Case log
- Open questions
- Current state
- Next action
- Escalation path
- Closure criteria
- Lessons learned
- Ruleset feedback

## Communication Standard

A well-formed case message should explain:

1. Who the person is and the relevant history.
2. What they are trying to accomplish.
3. What happened.
4. What has already been attempted.
5. What consequence the problem caused.
6. What specific resolution is requested.
7. What communication methods are workable.

The message should be built from the case record so the user does not have to reconstruct the problem under pressure.

## Closure Standard

A reply is not resolution.  
A referral is not resolution.  
A ticket is not resolution.  
A promise to investigate is not resolution.

Resolution means that the original desired outcome has been achieved and verified in practice.

## Version Control

Use semantic-style numbering:

- **Major version:** structural redesign or incompatible schema changes.
- **Minor version:** new capabilities, rules, phases, or schemas.
- **Patch version:** clarification, examples, wording changes, and corrections.

Every release should record:

- version,
- date,
- summary,
- rules added,
- rules changed,
- rules removed,
- reason for change,
- cases that exposed the need,
- known limitations.

## Initial Known Limitations

- The framework has not yet been tested across several types of cases.
- A case-management interface has not yet been created.
- Prioritization and escalation timing are not yet standardized.
- Professional judgment is still required for medical, legal, financial, and safety-critical matters.
- The first version may contain unnecessary steps or omit important ones.

## First Pilot Case

**Goal:** Obtain dental care.

The pilot should produce:

- a precise dental-care goal,
- a reconstructed history from existing records,
- a map of eligibility, referral, scheduling, and communication barriers,
- a roadmap with test points,
- a reusable case-interface concept,
- and improvement notes for version 1.1.

## Revision Philosophy

Version 1.0 is not intended to be complete. It is intended to be usable.

The ruleset should change whenever real use reveals:

- an unnecessary burden,
- an unclear rule,
- a missing field,
- a repeated failure pattern,
- an untracked handoff,
- a weak test point,
- or a better way to preserve the user’s context.

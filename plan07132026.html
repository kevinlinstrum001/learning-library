Single-Page Viewer Conversion Plan

A staged plan for replacing separate Apps Script views with one embedded dashboard that switches content internally and remains stable inside Google Sites.

Goal

Replace the four separate HTML pages with one embedded application that never changes the iframe URL.

Current structure: Cards.html WorkQueue.html RecommendationsQueue.html CompletedWork.html Target structure: Dashboard.html → Intake → Work Queue → Recommendations → Completed Work

Navigation will switch visible content with JavaScript rather than loading another page. The Google Sites embed will remain loaded and stable.

Keep the existing backend

The current Apps Script data functions can remain:

getSubmissionCards()
getWorkQueueCards()
getRecommendationsQueueCards()
getCompletedWorkCards()

The existing action functions can also remain:

acknowledgeSubmission()
sendToWorkQueue()
sendToRecommendationsQueue()
updateWorkItemStatus()
addWorkItemNote()
updateRecommendationItemStatus()
addRecommendationItemNote()
No spreadsheet structure needs to change. The conversion is primarily a front-end reorganization.

Create one dashboard file

Create a new Apps Script HTML file named:

Dashboard.html

Its basic structure will be:

<header></header>
<nav class="view-nav"></nav>
<main id="app-view"></main>

The app-view element becomes the shared display area for every section.

Simplify doGet()

function doGet() {
  return HtmlService
    .createHtmlOutputFromFile("Dashboard")
    .setTitle("Meadows Housekeeping")
    .setXFrameOptionsMode(
      HtmlService.XFrameOptionsMode.ALLOWALL
    );
}

The application will no longer depend on:

?view=work
?view=recommendations
?view=completed

Build internal navigation

<nav class="view-nav">
  <button data-view="intake">Intake</button>
  <button data-view="work">Work Queue</button>
  <button data-view="recommendations">Recommendations</button>
  <button data-view="completed">Completed Work</button>
</nav>

Each button calls:

showView("intake");
showView("work");
showView("recommendations");
showView("completed");

Central view controller

function showView(viewName) {
  setActiveNavigation(viewName);
  showLoadingMessage();

  switch (viewName) {
    case "intake":
      loadIntakeView();
      break;
    case "work":
      loadWorkQueueView();
      break;
    case "recommendations":
      loadRecommendationsView();
      break;
    case "completed":
      loadCompletedView();
      break;
  }
}
No links, redirects, targets, or iframe URL changes will be involved.

Convert each page into a render function

Intake

function loadIntakeView() {
  google.script.run
    .withSuccessHandler(renderIntakeCards)
    .withFailureHandler(showError)
    .getSubmissionCards();
}

Work Queue

function loadWorkQueueView() {
  google.script.run
    .withSuccessHandler(renderWorkQueue)
    .withFailureHandler(showError)
    .getWorkQueueCards();
}

Recommendations

function loadRecommendationsView() {
  google.script.run
    .withSuccessHandler(renderRecommendationsQueue)
    .withFailureHandler(showError)
    .getRecommendationsQueueCards();
}

Completed Work

function loadCompletedView() {
  google.script.run
    .withSuccessHandler(renderCompletedWork)
    .withFailureHandler(showError)
    .getCompletedWorkCards();
}

Every renderer writes into the same container:

const container =
  document.getElementById("app-view");

Track the active view

let currentView = "intake";

Whenever a note or status action finishes, reload the active view:

function refreshCurrentView() {
  showView(currentView);
}

Highlight the selected navigation button:

function setActiveNavigation(viewName) {
  currentView = viewName;

  document
    .querySelectorAll(".view-nav button")
    .forEach(button => {
      button.classList.toggle(
        "active",
        button.dataset.view === viewName
      );
    });
}

document.addEventListener(

Migration order

ABuild the shell

  • Create Dashboard.html.
  • Add header, navigation, content container, and loading message.
  • Test navigation buttons without loading data.

BMove Intake

  • Copy the intake renderer and actions.
  • Verify cards load.
  • Verify acknowledgment works.
  • Verify sorting works.
  • Verify persistent states remain.

CMove Work Queue

  • Copy the Work Queue renderer and actions.
  • Verify pending cards load.
  • Verify Start Work.
  • Verify notes.
  • Verify Complete.
  • Verify completed work leaves the active view.

DMove Recommendations

  • Verify unreviewed and reviewed records load.
  • Verify notes.
  • Verify review state.
  • Verify archive removes the card.

EMove Completed Work

  • Verify completed cards load.
  • Verify duration displays.
  • Verify notes display.
  • Verify completion email state displays.

FReplace the deployment page

  • Change doGet() to serve only Dashboard.html.
  • Update the Google Sites embed one final time.

Final testing

Test repeated navigation inside the Google Sites embed:

Open Intake → switch to Work Queue → switch to Recommendations → switch to Completed Work → return to Intake → repeat several times

Then test a full work lifecycle:

Submit form → acknowledge → sort to Work Queue → navigate to Work Queue → start task → add note → complete task → navigate to Completed Work → verify record and email state

Required results

  • The Google Sites page never reloads.
  • The embedded tool never disappears.
  • The iframe URL never changes.
  • Navigation works repeatedly.
  • The mobile layout remains usable.

Cleanup after successful conversion

Keep the older files temporarily as backups:

Cards.html
WorkQueue.html
RecommendationsQueue.html
CompletedWork.html

After the dashboard has completed several real work cycles, rename or remove them:

Legacy_Cards.html
Legacy_WorkQueue.html
Legacy_RecommendationsQueue.html
Legacy_CompletedWork.html
Leave the legacy files in the project until the new dashboard has been stable through several real submissions, acknowledgments, queue actions, completions, and recommendations.
